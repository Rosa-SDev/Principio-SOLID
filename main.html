Principio de Responsabilidad Única (SRP)

El principio de responsabilidad única establece que una clase debe tener una única razón para cambiar, lo que significa que debe tener una sola responsabilidad o función. Esto ayuda a mantener el código más modular, fácil de entender y de facil mantenimiento.

¿Qué significa?

  • Cada clase hace una sola cosa.
  • Si se necesita cambiar algo, solo tocas una clase.
  • Tu código es más fácil de entender y mantener.

Ejemplo en Python

Mal - Sin SRP (una clase haciendo todo)


class UserManager:
    def create_user(self, username, password):
        # Lógica para crear un usuario
        pass

    def delete_user(self, user_id):
        # Lógica para eliminar un usuario
        pass

    def send_email(self, user_id, message):
        # Lógica para enviar un correo electrónico
        pass

Bien - Con SRP (clases separadas para cada responsabilidad)


class UserManager:
    def create_user(self, username, password):
        # Lógica para crear un usuario
        pass

    def delete_user(self, user_id):
        # Lógica para eliminar un usuario
        pass

class EmailService:
    def send_email(self, user_id, message):
        # Lógica para enviar un correo electrónico
        pass

Cómo aplicarlo

Preguntas para detectar violaciones

01
¿Tu clase hace más de una cosa?

Si puedes describir la clase usando la palabra "y" (guarda datos y envía emails), probablemente viola el SRP. Divide la clase.

02
¿Cuántas razones tiene para cambiar?

Identifica quién puede pedir un cambio: ¿el equipo de base de datos? ¿el equipo de emails? Si la respuesta incluye múltiples actores, separa la clase.

03
¿El nombre describe exactamente lo que hace?

Si el nombre de tu clase o función es vago (Manager, Handler, Processor), suele ser señal de múltiples responsabilidades mezcladas.

04
¿Es difícil escribir pruebas unitarias?

Si una clase es difícil de testear en aislamiento, generalmente es porque tiene demasiadas dependencias y responsabilidades. El SRP facilita el testing.

Buenas prácticas

  • Dale a cada clase un nombre claro que describa una sola responsabilidad.
  • Si una clase tiene más de 2 o 3 métodos muy distintos entre sí, considera separarla.
  • Agrupa el código por quién lo usa, no por conveniencia.
  • Prefiere muchas clases pequeñas y específicas sobre pocas clases grandes.

Beneficios

  • Mantenimiento: Facilita el mantenimiento del código.
  • Legibilidad: Mejora la legibilidad y comprensión del código.
  • Reutilización: Permite una mejor reutilización de clases.
  • Pruebas: Facilita la escritura de pruebas unitarias, permite hacer test de manera unitaria y aislada.
  • Escalabilidad: Hace que el código sea más fácil de escalar y modificar sin afectar otras partes del sistema.
  • Colaboración: Facilita la colaboración entre desarrolladores al tener responsabilidades claramente definidas.
